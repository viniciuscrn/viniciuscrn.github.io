1. Sistemas Proprietários × Livres

1.1 Sistema Operacional Proprietário (Fechado)

Um sistema operacional proprietário é desenvolvido e controlado por uma empresa ou organização. O código-fonte não é disponibilizado ao público, e seu uso é regulado por uma licença comercial que proíbe redistribuição e modificação sem autorização.

Características principais:

  • Código-fonte inacessível ao usuário final
  • Distribuição controlada pelo fabricante
  • Suporte oficial fornecido pela empresa desenvolvedora
  • Personalização limitada ou inexistente pelo usuário
  • Auditorias de segurança realizadas apenas internamente (caixa-preta)
  • Geralmente exige pagamento de licença

Exemplos:

  • Microsoft Windows (10, 11, Server)
  • Apple macOS / iOS / iPadOS
  • IBM z/OS (mainframes)
  • Oracle Solaris

1.2 Sistema Operacional Livre / Open Source

Um sistema operacional livre disponibiliza seu código-fonte publicamente. Qualquer pessoa pode estudar, modificar e redistribuir o sistema, desde que respeite os termos da licença associada (GPL, MIT, BSD, Apache etc.).

Características principais:

  • Código-fonte disponível e auditável por qualquer pessoa
  • Comunidade global de desenvolvedores contribui com melhorias
  • Suporte oferecido pela comunidade e por empresas terceiras
  • Altíssima capacidade de customização
  • Auditorias de segurança públicas e contínuas
  • Geralmente gratuito (mas não obrigatoriamente)

Exemplos:

  • Linux (Ubuntu, Fedora, Debian, Arch, etc.)
  • FreeBSD / OpenBSD / NetBSD
  • Android (AOSP — Android Open Source Project)
  • Haiku OS

1.3 Quadro Comparativo

Critério Proprietário Livre / Open Source
Custo de licença Geralmente pago Geralmente gratuito
Acesso ao código Negado ao usuário Disponível publicamente
Suporte oficial Fornecido pelo fabricante Comunidade + empresas terceiras
Customização Limitada ou nula Total (dentro dos termos da licença)
Auditoria de segurança Interna (caixa-preta) Pública — qualquer pesquisador pode auditar
Uso típico Corporativo e desktop doméstico Servidores, embarcados, supercomputadores

⚠️ Atenção: “gratuito” ≠ “livre”. Um SO pode ser gratuito e proprietário (ex.: versões educacionais do Windows) ou pago e livre (ex.: Red Hat Enterprise Linux com suporte comercial).

2. Monotarefa × Multitarefa

2.1 Sistema Monotarefa

Em um sistema monotarefa, apenas um programa pode ser executado de cada vez. Enquanto um processo está em execução, nenhum outro pode ser iniciado. A CPU permanece ociosa durante operações de entrada/saída (I/O), o que representa grande desperdício de recursos.

Características:

  • Arquitetura simples — sem escalonador de processos
  • Um único programa ocupa toda a memória de usuário de cada vez
  • CPU fica ociosa durante leituras de disco ou espera de dispositivos
  • Sem necessidade de isolamento entre processos

Exemplos históricos:

  • MS-DOS
  • CP/M
  • Firmware de microcontroladores simples (ex.: Arduino sem RTOS)

2.2 Sistema Multitarefa

Em um sistema multitarefa, o SO gerencia múltiplos processos “simultaneamente”, alternando rapidamente entre eles (time-sharing) ou executando-os em paralelo real em múltiplos núcleos de CPU.

Características:

  • Escalonador de processos distribui o tempo de CPU
  • Isolamento de memória entre processos é essencial
  • CPU raramente fica ociosa — outro processo assume durante I/O
  • Exige mecanismos de sincronização (semáforos, mutexes etc.)

Exemplos:

  • Todos os SOs modernos: Windows, Linux, macOS, Android, iOS

2.3 Tipos de Multitarefa

Multitarefa Preemptiva

O SO interrompe processos automaticamente por meio de um temporizador (timer interrupt), independentemente da vontade do processo. É mais robusta: um processo travado não bloqueia o sistema inteiro.

  • Usada em todos os sistemas operacionais modernos
  • Garante melhor responsividade e estabilidade

Multitarefa Cooperativa

Cada processo cede voluntariamente a CPU ao sistema quando não precisa mais dela. Se um processo não ceder (por bug ou má-fé), trava o sistema inteiro.

  • Usada no Windows 3.x e no Mac OS clássico (antes do OS X)
  • Modelo frágil — abandonado em SOs de uso geral

2.4 Quadro Comparativo

Critério Monotarefa Multitarefa
Complexidade do SO Baixa Alta (escalonador, memória virtual…)
Aproveitamento de CPU Ineficiente (ociosa em I/O) Eficiente (outro processo assume)
Isolamento de processos Desnecessário Essencial — evita interferência
Responsividade Baixa — um processo por vez Alta — múltiplas tarefas em paralelo
Casos de uso atuais Microcontroladores simples, IoT Desktops, servidores, smartphones

3. Monousuário × Multiusuário

3.1 Sistema Monousuário

Um sistema monousuário é projetado para atender a um único usuário de cada vez. Não há controle de acesso elaborado — todos os recursos pertencem a quem está usando a máquina no momento.

Características:

  • Sem autenticação obrigatória (ou autenticação básica)
  • Sem isolamento de dados entre perfis de usuário
  • Sem controle de permissões entre sessões
  • Arquitetura mais simples de gerenciamento de recursos

Exemplos:

  • MS-DOS
  • Windows 95 / 98 / Me
  • iOS e Android (tecnicamente monousuário — apenas um perfil ativo por vez)

3.2 Sistema Multiusuário

Um sistema multiusuário permite que vários usuários utilizem o sistema simultaneamente — de forma local ou remota (via rede, SSH, Terminal Server etc.). O SO deve isolar recursos, gerenciar permissões e autenticar cada sessão de forma independente.

Características:

  • Autenticação individual obrigatória
  • Controle de acesso por usuário/grupo (permissões rwx no Linux, ACLs no Windows)
  • Isolamento de memória entre sessões
  • Gerenciamento de cotas de disco e CPU por usuário
  • Suporte a sessões simultâneas (locais ou remotas)

Exemplos:

  • Linux / Unix (qualquer distribuição)
  • Windows Server (com RDS — Remote Desktop Services)
  • macOS (baseado em Unix — suporte nativo a multiusuário)
  • IBM z/OS

3.3 Recursos que o SO Multiusuário precisa oferecer

Recurso Descrição
Autenticação Identificação única por usuário (senha, biometria, token, LDAP/AD)
Controle de acesso Permissões de leitura, escrita e execução por arquivo, diretório e recurso
Isolamento de memória Processos de um usuário não acessam o espaço de memória de outro
Cotas de recursos Limite de CPU, memória e disco por usuário ou grupo
Auditoria Registro de ações por usuário (logs de acesso, modificações etc.)

ℹ️ Nota sobre Android e iOS: Ambos são tecnicamente monousuário — apenas um perfil está ativo por vez. O Android oferece suporte a múltiplos usuários em tablets, mas não com sessões verdadeiramente simultâneas como no Linux.

5. Tendências Atuais

Convergência das categorias

SOs modernos são quase universalmente multitarefa + multiusuário. A distinção “mono × multi” tornou-se relevante principalmente em:

  • Sistemas embarcados e IoT (ex.: firmware de roteadores, Arduino)
  • Dispositivos de uso muito específico (ex.: caixas eletrônicos com SO dedicado)

Open source no mercado corporativo

  • Mais de 96% dos servidores web do mundo rodam Linux
  • 100% dos 500 supercomputadores mais rápidos do mundo usam Linux
  • Empresas como Microsoft, Google e Meta contribuem ativamente com projetos open source
  • Windows adotou subsistema Linux nativo (WSL 2) em 2019

Modelos híbridos

  • Android é open source na base (AOSP), mas os serviços Google (GMS) são proprietários
  • macOS é baseado no kernel Darwin (BSD — código aberto), mas o sistema como um todo é proprietário
  • Red Hat Enterprise Linux é código aberto, mas vendido com suporte comercial
